Swift 3.0 GET and POST

by Micheal Willard

Section 1: Introduction

This Tutorial will go over the basics of making GET and POST requests to a URL based API using Swift 3.0. A number of changes have happened between Swift 2.x and Swift 3.0, so there isn’t a wealth of documentation available currently. Fortunately, most of the deprecated functions names and conventions are recognized by Xcode. Xcode will often provide you with the ability to update your code to the latest syntax and naming convention.

In order to simplify the tutorial, we will be taking some of the focus off the iOS interface creation. To do this, we will base the project on the Apple Developer site’s “Start Developing iOS Apps (Swift) tutorial.” It is recommended that you work through that project prior to implementing the GET and POST calls. Additionally, the iOS development terminology used in the tutorial will be more familiar to you.

This tutorial will show you how to build a table view app that allows you to add an item to the table, view the table and select an item in the table to view details. This app saves the Objects locally, whereas we want to save them to a cloud database via POST calls and retrieve the objects via GET calls. But it gives us a good template interface for making these actions.

The API used in this tutorial, again, is URL based. It is a personal API developed for a specific functionality. In your usage, you can simple swap out the URL strings to make GET and POST calls to the API of your choice. Section 2 will go into more depth on the API, to give you a better understanding of how to implement your own GET and POST calls into your project.

This How-To will walk the you through the following items:

  • Section 1: Introduction
  • Section 2: App and API Overview
  • Section 3: GET
  • Section 4: POST

As discussed in the Introduction, the API is a custom built backend. The GET and POST calls are URLs that return JSONs. This JSONs can then be manipulated into the data you display in your app. This is a common format for a lot of web API, so your approach should be similar enough that you can adapt these methods.
